# 📝 好學生筆記：非技術人員如何用 Claude Code 統治工作流

> **學習摘要**：本筆記整合了 Zeabur 創辦人關於 Claude Code 的深度實踐分享與 QA 實錄。核心教學包含：如何透過「對話」Dump 大腦思維以生成 Web 簡報、讓 PM 獨立驅動數據可視化與 Discord 自動化機器人。此外，更介紹了「斑馬手冊」知識庫概念，透過 GitHub 實現 Local First 的 AI 知識同步與踩坑經驗共享。最後，強調將產品「Skill 化」以迎接 Agent 執行者時代的到來。

---

# 第一部分：演講實錄

我希望今天有這個機會可以跟大家分享。對，那我的...稍等一下。

## 現場互動與 Meetup 背景

好，那他們在處理技術問題的時候，我先來做一些小小互動好了。因為大家知道這一次的 Claude Code Meetup，因為人數太多的關係，我們在台北分了三個場次，分別是工程師，然後我們這邊是 PM & Designer，還有一場是做 Business 跟 Operation。我想問一下今天到場的朋友們，哪些人是 PM？可以舉手嗎？OK，大概一半，蠻不錯的。那誰是設計師？UI, UX 之類的？也是剛好，好像感覺剛好一半。有沒有不是 PM，也不是設計師的？比如說有沒有人是，欸，那同學，你不是，那你是什麼？那你是工程師。啊，你是工程師，那你好像有點走錯場囉。對，工程師今天是在那個中研院那邊。

那是沒關係，因為我其實也想了很久，就是到底我應該用什麼樣的內容來分享。然後我就覺得說，畢際 Claude Code 它本來叫做 Code，可是其實它已經遠超過可以只是做 Coding 的範圍了。然後我就想分享一些是從，因為我們現在在創業嘛，在做 Zeabur 這個產品，所以我就想要從一個 Founder 的角度，去分享說到底 Zeabur 現在除了拿 Claude 來寫程式之外，我們還做了什麼。

我們不只是鼓勵員工用 Claude Code 來做他們的開發工作，我們也鼓勵非技術、非工程師的夥伴，也用 Claude Code 來做他們的日常工作。但做什麼呢？你們一定會想說，阿我不是工程師，怎麼要買 200 美金的 Claude Code？我們等一下就來分享四個理由。對，不過我要先講的是，不是說真的每個員工一進來就發 200 美金啦。是還要證明他用得到 200 美金，否則一般的話可能就先 20 美金。然後像剛剛分享的這個 Lin，他就是 20 美金用了大概兩天吧，發現，欸，不夠了，我要 100，他就跑來升級，然後就升級到 100 美金。現在離 200 美金還有一點路要走，所以我們會再跟他討論一下這個部分怎麼優化。

> 🎯 **產品經理 (PM) 視角**
>
> - **類比**：以前 PM 是「翻譯官」，負責把客戶的需求翻譯成工程師聽得懂的語言；現在透過 Claude Code，PM 變成了「自帶工廠的設計師」。
> - **應用**：在預算管理上，應將 200 美金的訂閱費視為「技術槓桿」而非「工具支出」。如果一位 PM 能透過 AI 獨立完成數據儀表板，這筆投資的 ROI (投資報酬率) 極高。
> - **連結**：與「垂直領域專家 + AI 能力」的複合型人才趨勢關聯。

## 結合 Claude Code 的四個姿勢

那我今天想分享的 Zeabur 在團隊內部去結合 Claude Code 的四個姿勢，分別是這四個：做簡報、讓非技術人員參與開發、Agentic AI 夥伴，以及將產品結合到產品中。

### 姿勢一：用 Claude Code 做簡報

做簡報，讓非技術人員也可以上手參與團隊內部的一些，可能不一定是產品本身的開發工作，但是可能跟產品有相關性的工作，他們也可以來參與。我把 Claude Code 當成一個 Agentic AI，所以它不只是拿來寫程式而已，它甚至變成是我們的一個夥伴一樣的方法，跟我搭配著一起工作。把這套產品結合到產品中，並不只是幫你一起做產品，更多成為產品的一部分，或是產品體驗的一部分。我今天分享的主要是圍繞這四個姿勢展開。

用 Claude Code 做簡報，大家可以看到簡報右上角有寫，簡報是 Claude Code 製作的，是今天早上做的，大概花了半個小時就做好。這邊就是今天早上我熱騰騰剛截圖的，我這個簡報的剛開始的一個截圖。它有點小，我看能不能放大。其實我的 Prompt 很簡單，就是我跟他說，前面是我自己的技術偏好，你們如果不知道那是什麼可以跳過。反正就是說，幫我在這個 Claude Code 的 Meetup 大會做了一個簡報。然後我想要風格是怎麼樣，然後主題色是什麼，然後我大概要分享的內容是什麼，他其實就開始寫了。對，那開始寫之後其實，就很快很快他就把一個大概的架構先搭出來了，搭出來以後，剩下的下一步其實才是最精彩的地方。

為什麼我用 Claude Code 做簡報？因為我知道很多人可能會喜歡用 Gamma，如果你要用 AI 做簡報，可能是 Gamma，可能是 Gemini，現在也能用 Google 來做。但是我覺得我最喜歡的一個點是，我喜歡用對話來做簡報。因為我覺得我自己是一個思維，就是想東西想很多，可是總是沒有辦法很好地去用簡報來把它表達出來的人。包含像如果有人 follow 我的 Threads 的話，你會發現我的寫文章的能力非常糟糕。只是我之前分享了一篇是介紹各種技術名詞嘛，那其實我只是把我想到了什麼，我就馬上寫下來。它其實整篇可能不是很有架構，或者是用語沒有很精煉的一個東西。

但是當我今天跟 Claude Code 在對話的方式做簡報的時候，其實它不是像 Gamma 一樣，把我貼進去的內容變成一個簡報，而是我先跟它講我在想什麼，他搞清楚我在想什麼以後，重新整理一次，到底我想表達的是什麼，然後再把它用一個正確的架構做出來。所以，其實我會下一個結論是說，做簡報的本質，它是在 Dump，就是把我大腦裡面的內容搬出來的一個過程。而把大腦搬出來的這個過程，其實本質上用對話的方式是最天然的，最自然的。就好像我跟朋友聊天，聊完之後他如果真的能 Get 到我在講什麼的話，他來幫我做簡報，那就會比我自己很爛的表達能力來做來得更好。對，所以這個是為什麼我會現在很喜歡用 Claude Code 來做簡報，而不是 Gamma 的原因。

對，那這邊其實除了剛剛提的之外，還有一個問題是，我覺得現成的大部分的 AI 做簡報的工具或者是軟體，它都存在一個問題是，他們都是很早以前就開始做這個產品了嘛。當時的 AI 其實沒有現在的模型來得這麼厲害，所以他們在產品的初稿，或是整個架構在設計的時候，他為了要讓那個沒有這麼聰明的模型，可以做出一個勉強能接受的簡報，他其實會大量的依賴一種叫做模板的東西。就是 AI 先搞懂我要幹嘛，然後找出適合套哪個模板，把我的內容套進那個模板。所以不管你今天是 Gamma，還是 Gemini，還是一些其他東西，它做出來總是給我一種非常重的模板感，那個東西是我非常不喜歡的。

可是， Claude Code 的這個工具，它其實本身開發出來不是為了做簡報，它是為了可以開發任何東西。所以在這個情況下，它其實只是用一個可以開發任何的東西來剛好開發的一個簡報。所以其實這個簡報的本質上它沒有套任何的模板，它是用無限的可能，你想做什麼就可以跟他講，然後他就幫你做。所以除了我剛剛講的第一個是把你的對話的方式把概念做成簡報之外，其實你是在寫網站。你寫網站的話，在這個網站裡面可以加任何東西。像這邊就是說，你想要加什麼酷炫功能都可以，然後這句話它就是可以按的。按按鈕會噴彩帶，雖然不知道為什麼簡報可以噴彩帶，但我只想要加一個這個東西。

最重要的還有一個點，現在我相信來這個活動的所有人，應該都有至少付 20 美金給 Claude Code，啊你都已經付 20 美金了，你就不要再去買什麼 Gamma 之類的東西了。20 美金它是包含了做簡報的功能，所以你就可以繼續用同樣的工具來開發，不需要額外付錢。我剛跟他用對話的方式，說我要放大約...我要寫做什麼酷炫功能都可以，你就幫我做個酷炫功能進去好了。他就幫我做了一個噴彩帶的功能，這就是我剛想要表達的兩個概念結合起來：一個是用對話的方式，我想到什麼，我跟他講，他來幫我做。

大家知道有在用 Claude Code 都知道有一個技巧的東西，如果我把這個簡報，跟我做出這個簡報的過程，變成一個技巧的話，那其實我可以把這個技巧分享給 Zeabur 的其他人。所有員工當他們要去外面分享跟 Zeabur 有關的東西的時候，他就可以跟 Claude Code 對話，他就不用從頭開始講這麼多東西了。他其實會從我今天已經跟 Claude Code 對話出來的經驗裡面，抽取出可以重複使用的部分。然後直接做出一個，比如說在 Branding，或者在排版、在元件，或者是酷炫功能的部分，他就可以用我以前用過的東西，直接套進他們要做的那個新的東西。但是裡面的內容是那個員工他想要分享的東西，把它放進去我們的...再來第二個點是什麼？就是用這樣子的方法以後其他人就不用像我今天一樣這麼痛苦的從零開始。Logo 左上角、右邊放其他品牌等，這些小東西可以直接使用 Skills 去使用。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：傳統 AI 簡報工具是「微波爐食品」（加熱模板）；Claude Code 則是「私廚」（根據對話需求從食材開始開發）。
> - **應用**：PM 在進行 User Story 的腦暴時，可以直接將對話紀錄餵給 Claude Code，生成一個可互動的產品原型 (Proto-Presentation)。
> - **連結**：與「設計代碼化 (Design to Code)」與「無頭內容管理 (Headless Content)」關聯。

### 姿勢二：數據可視化與流程自動化

第二個內容是，我們的內部如何讓這些非技術人員搭配 Claude Code 使用。我想分享的主要是兩大場景：第一個是數據的可視化，第二個是流程的自動化。這兩個詞其實大家都可能都很了解，因為以前有一個東西很紅過一陣子叫 N8N。N8N 其實本質上就是說，我不會寫程式，但我用堆積木的方式堆一堆，公司裡面有一些工作，它就可以被自動化的去重複使用。數據可視化這個事情的話比較像是，我不知道有沒有 PM 有玩過叫做...Google 有一個叫什麼來著？Looker Studio。

Looker Studio 它可以幫你從資料庫裡面把東西拉成圖表，類似像那樣子的東西。或者是可能有 GA 的數據，然後你們如果用 PostHog，可能有 PostHog 的數據。散落在各地的數據，以前都會依賴可能要 PM 加工程師的方式，來幫你拉出一個可以看的 Dashboard。現在的話我們的團隊內部是這樣子。我們現在的數據可視化，就是以前是 PM 想要，舉個例子喔：PM 想要查看 Zeabur 在上個月，各個品牌的伺服器銷量的一個情況。那因為 PM 他只懂數據面的東西跟產品面的東西，他不懂資料庫是什麼，他不懂資料庫裡面有什麼。所以他就要去找工程師說：「欸，那個你可以有空的時候幫我拉他一個表格出來嗎？然後我這個 Excel 我自己再去外面畫那個圖就好了。」可是每個月、每個禮拜都要去麻煩工程師，工程師也會覺得說：「那我開發工作都做不完了，我還要幫你弄這個東西。」就很累。

可是現在的話，PM 自己用 Claude Code，它可以在我們的 Admin 後台管理系統裡面，自己加一個 Dashboard。那那個 Dashboard 裡面要放什麼東西，完全它自己決定。它想要發什麼數據，完全它自己跟 AI 溝通完之後，AI 告訴它可行或者是不可行。不可行的話為什麼不可行，為什麼不可行的話我要找誰才能讓它變得可行。這些問題全部都是 Claude Code 自己告訴它，PM 它的能力範圍就會變得很大很大。

實際案例是我們團隊內部真實的一個圖表。如果這邊有 Zeabur 的用戶的話，應該有認識潘哥，潘哥是我們的元老級的工程師。他有一天突然在我們的產品頻道貼了一個我完全沒有看過的圖，然後說這個是 Vita 做出來的東西。然後就有六個員工按了一個牛的符號在下面。那當時看到這個東西的時候，我是整個嚇壞。因為這個東西是我一直跟...哦，Vita 是我們家的 PM。我不知道 Vita 今天有沒有來？Vita 沒有來，OK，她可能今天放假。對，那就是我一直跟 Vita 說，我好想要知道我們現在...因為剛出了那個專用主機的功能，我想要知道到底哪個品牌賣得最好，然後成長的幅度是怎麼樣。因為我要知道這些數據，我們才知道下一步要怎麼做嘛。

我就一直跟這個 PM 說，我們可不可以搞一些什麼方法，然後以前他就要一直跟工程師要數據。突然有一天，他就自己在我們的 Admin 加了一個這個整個監控圖表。而且監控圖表是結合了他在 PM 領域的一些數據上的背景知識，去設計出有一些圖表是連我都沒有想到的。比如說用這種堆疊圖的方式來去呈現。其實我可以更好地理解每天跟前一天的銷量比例。可是在對疊圖裡，我又可以同時看到每個品牌之間的比例是怎麼樣的。等於說我只懂商業面的東西，可是它作為 PM 更懂數據面的東西。它把它的東西結合起來之後，反而幫我們製造出了一個新的可以看數據的方法。對，所以這是第一個 Case。

然後後面馬修，馬修是我們另外一位工程師，他就意外的發現，它不只做了這個圖表，因為 Zeabur 有一個服務叫 AI Hub，就是專門是，比如說我可能有一天想要用 Gemini 的模型，明天想用 GPT 模型，後天又想用 Claude 模型，AI Hub 把模型結合在一個 API，讓你可以統一調用。Vita 每天都有新用戶問 GPT 的新模型什麼時候上，或是有人想問很冷門的模型，譬如說千問模型、GLM 模型為什麼沒有。有天他就意外的發現，Vita 已經從 Claude Code 習得了在 AI Hub 新增上架新模型的能力。這是馬修他在我們的 Git 裡面發現的，一個很意外的事情。所以在這個截圖的案例，我們想要分享的是說，當一個 PM 或者說一個非技術背景的同仁，掌握了 Claude Code 他的能力範疇以及使用方法以後，他其實可以把他的能力邊界擴張到一個其他人都無法想像的程度。

第二個 Case 是流程自動化。流程自動化當然有很多不同的接入點。以 Zeabur 來說，因為我們的客戶的 Community 還有我們自己內部的管理的方法，都是用一個叫 Discord 的 App 來去做管理。做一個 Discord 的機器人是最簡單的接入方法。一樣是 Vita，一開始還不知道是 Vita。一開始我們在頻道裡面講幹話，講一講。突然機器人衝出來，把我們上週的累積註冊用戶、付費用戶、營收情況、各個產品的情況，突然就甩了一句這樣的東西出來。我就說：「哇，好屌，這誰做的？」密碼就說又是 Vita 做的。真的非常意外。這讓我覺得你們臨場會看到的東西就是 Zeabur 對工作的這種...就是我可以去提醒某個 Project Owner 說：「你的專案怎麼已經兩個禮拜沒更新了，是不是要 GG 了？趕快給點 Update。」這樣子。那所有東西都是沒有工程師參與，完全用 Claude Code 就直接開發完成，直接上架的一些內部工具。

在第二個姿勢裡面的一些 Takeaway，我覺得 Claude Code 一開始是作為一個 Coding 工具沒錯，但它後來越來越發展，它的生態越來越完整以後，它其實現在已經足以讓非技術背景的人能夠獨立上手。當然有些人可能會抱持懷疑態度，因為如果你去網路上搜一些比如說 Coding 的課程，你在跟他聊天然後寫程式，那其實根本就不是 Claude Code，那是 Claude 的那個聊天的 App。所以，光是你要搞懂什麼叫 Claude Code，然後我應該怎麼安裝 Claude Code，怎麼開始使用 Claude Code，就已經遠高於這個 Google AI Studio 或者是其他的門檻了。因為這個門檻的關係，很多人踏不進我們現在正在體會到的這個神奇的事情。但是老實說，在你的 MAC 上面打開 Terminal 是一個難到我花一個月都學不會的事情嗎？其實不會吧。等一下 Demo 30 秒，其實大家都可以學會怎麼開 Terminal。在 Terminal 裡面輸入 Claude 這個單字是一個很難的事情嗎？其實好像也不是。對，所以我們現在正在想要把這個東西推出去，就是所有即使你覺得你是個小白，我也要想要讓你相信你是有辦法玩轉 Claude Code 玩起來的這件事情。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：以前 PM 是「在岸邊看魚的人」，現在是「拿著 AI 魚叉潛水的人」。PM 以前只能看工程師產出的報表，現在可以直接進入產品的 Admin 層進行開發。
> - **應用**：PM 應爭取內部管理系統 (Admin Portal) 的低門檻修改權限，透過 Claude Code 實現「數據民主化」，不再受限於工程師的排期 (Sprint)。
> - **連結**：與「自助式分析 (Self-service Analytics)」與「運維自動化」關聯。

### 姿勢三：團隊共用的 Agentic AI

第三個，我們 Zeabur 內部在用 Claude Code 的一個姿勢，叫做團隊共用的 Agentic AI。說到 Agentic AI 這個詞，或者說一個可以自主執行任務、自主完成任務的 AI Agent，會想到什麼名詞？有沒有人有？龍蝦。其實 Agentic AI 這個詞，好像過去一個月龍蝦的話題性會更高。但是我們團隊內部現在是把 Claude Code 當作這樣子的定位，而不是用龍蝦來去做。

這邊我也來分享我們的一些經驗跟我們的 Takeaway 是什麼。其實我們自己觀察下來，因為我們肯定是大家都有玩過。如果你之前想要在公司裡面用，量比較大，或者是穩定性、安全性要求比較高，其實我們現在還是主推 Claude Code 這一端。背後技術來分享，為什麼會有這樣子的一個想法？其實本質上，我覺得兩個東西最大的差異點是 Open Claude 它能做任何事情，只要你給它一台 Mac mini，用你自己的電腦能做的事，它都能幫你做。譬如說訂房、訂機票、看股票等等的這些個人的使用場景，他可以直接開一個瀏覽器，或者是直接開一個 APP 來幫我操作。可是如果他沒有 API、沒有 CLI 或者是沒有 MCP 的話，其實 Claude Code 基本上就不用想了，他就是完全做不到，他只能鬼打牆。

所以，在這個情況下，Open Claude 你可以說他的能力其實更強。但是更強的同時，他是犧牲掉了什麼換來這個更強的能力？犧牲掉的是它的效率跟它的成本換出來的。比如說為什麼它能夠直接操控我的電腦？是因為它是直接截圖我的電腦，然後把它丟進大模型。知道說，哦，為什麼...欸？有變紙嗎？好，我們處理一下。你幫我操作一下。對。小龍蝦它其實本身上截圖完了之後，並使用截圖的上傳...Token 量非常非常大。而 Claude Code 它指定裡面就是一點點的文字過去，一點點的文字回來，Token 量非常非常少，而且它的效速度也非常非常高。所以在這兩個東西，我們其實現在的一個比較下來是這樣子。所以我們也不會說直接就是踩一捧一，說小龍蝦不好，然後 Claude Code 很讚，我們也不會這樣子講。其實還是看到底你現在的需求是什麼，你想做的事情誰能做到，誰做得更快、誰做得更便宜，然後我們來去推進你做這個東西。

那其實你也可以把它兩個東西一起用。這個是我們剛剛在 setup 這個簡報的時候，我們的那個豆腐同學，他突然就秀一下操作，然後把我震驚到了。就是因為剛剛他們說，突然說簡報要加一頁，是可以加一個大合照的頁面。然後他就突然就跑去跟他的小龍蝦說：「你那個簡報幫我加一個 PicCollage 的照片進去。」然後加完之後我們要拿來去做什麼掃碼的環節。然後他的小龍蝦就嗶嗶嗶嗶嗶，然後直接更新到他線上的 Zeabur 上，Zeabur 上的他的那個線上簡報連結上面去。這整個 Workflow 本質上是豆腐讓小龍蝦去用 Claude Code 來開發，並調用 Zeabur 的 Skill 去完成部署的一整個鏈路。所以，其實你會發現當你真的參透了這幾個 AI 工具它的本質跟它的實現原理以後，他們是可以互相搭配使用的，而不是一定要說我就是要用小龍蝦、我不要用 Claude Code，或者反過來。

不過其實老實說，用龍蝦調 Claude Code 好像是一個有點灰色的一個地方啦。因為 Claude Code 的本質上是不允許用非人類的方法去使用它的。如果你真的用得非常的兇的話，可能 Anthropic 官方會封禁。所以作為 Claude Code 官方的社群活動，還是呼籲大家，用官方建議的使用方法去使用這些工具會比較好。已經有許多人的 Key 被鎖掉了，大家玩一下要小心喔。

第三個點，我們現在在聊的是什麼？是聊到說讓技術背景的人，也能在團隊裡面用 Claude Code 參與一些事情。這個地方我們就想要分享， Zeabur 在上個月推出了一個內部的概念，叫做「斑馬手冊」。什麼叫斑馬手冊呢？其實就是在玩梗。因為 Zeabur 的名字很多人以為是 Zebra，就以為我們公司是斑馬。我們就接受這個設定，研發了一個叫斑馬手冊。它其實是一個 Knowledge Base，但我們沒有用 Notion，也沒有用其他的什麼非常花裡胡哨的設定。我們就用一個 Github 的 Repo，開了一個很單純的 Github 的倉庫。讓所有人，或者是所有 AI Agent，把你知道的所有關於 Zeabur 的東西，全部砸進這個 Repo 裡面，也不用去整理。

這次想要做到的效果是什麼呢？下面有一個舉例，當然這個舉例是非常限縮的一個小舉例。大家都知道 Zeabur 有一個論壇功能，然後我們的論壇會有很多的 Support Engineer，我們的客服工程師會去回答大家。可是今天如果有一個人他收到了問題以後，他必須要靠自己的知識去回答的話，那知識是很有限的，人的知識是很有限的。所以他現在就可以讓 Claude Code 去斑馬手冊裡面看，有沒有人回答過這個問題，或者是關於這個問題他的一些 Background 的知識是什麼。Claude Code 找到了以後，推理一下然後告訴我們：「欸，以前有人已經回答過這個問題，寫在斑馬手冊裡面了。所以你這樣子回就好了。」

可是也有另外一種情況是，欸，斑馬手冊裡面查不到，對吧？查不到的時候會發生什麼事情呢？因為我們在它的 Memory、Global Memory 裡面寫了一句話是：如果你試圖查詢斑馬手冊，結果你要查的東西在裡面查不到的話，你有一個義務是去那個 Repo，就 GitHub 的那個 Repo，開一個 Issue 說「我想查一個什麼東西，但我沒查到」。這個 Issue 一旦開上去以後，就會有人的工程師，或者是人的工程師的 AI，進去裡面看這什麼情況，為什麼有一個關於 Zeabur 的問題沒有在斑馬手冊裡面，然後我們就會在越短的時間內去把它補進去斑馬手冊。用這個方法，如果把這個東西嚴格地跑個，比如說一個月啊、兩個月，可能都不用這麼久，可能兩個禮拜。其實你的斑馬手冊會變得非常的完整。所有真的關於 Zeabur 或是關於你的公司，它就變成一個百科全書。對。那這個百科全書有一個什麼好處呢？其實不只是論壇就是客服的場景可以用，在新員工 Onboarding 的時候也非常好用。他有任何關於你們公司的問題，可是他太菜了，他不懂，還有不好意思問前輩，怎麼辦？問 Claude Code。Claude Code 幫他翻斑馬手冊，就翻出來說你應該要怎麼樣怎麼樣的。

所以，其實我們現在想要講的是，斑馬手冊的這個例子，它其實有點沿用了以前我們在講的一個 RAG 的概念嘛。就是模型本身它其實只知道模型的訓練數據。但若掛一些額外的知識庫進去，當作 RAG 去幫他檢索回來的話，它其實能力的範疇會變得更強。現在我們想要讓這個斑馬手冊變成團隊所有人的 Claude Code 共享一份知識庫。它不是團隊所有人共享一份知識庫，是團隊所有人的 Claude Code 共享一份知識庫。因為團隊現在所有人都有 Claude Code，公司已經買給他們了。他們的工作其實也不是自己在做，也是他們的 Claude Code 在做。就好了，人也不需要去看那個知識庫了。

對，所以我們用這樣子的方法，我們不只把 Zeabur 內部的技術架構知識化了以後寫進去這個手冊裡面，我們同時還可以把踩坑經驗也共享起來。而且踩坑經驗不是人的踩坑經驗，是 Agent 的踩坑經驗。因為 Claude Code 他在做東西的時候，即使他真的花了半小時跑完，其實你如果願意花時間進去看的話，他每個步驟可能都在踩坑。比如說我打這個指令結果失敗了，失敗以後她跟我說這邊要怎麼樣？好，我再試試看。其實她如果能夠把這個經驗記錄起來的話，下一次她就不用花半個小時，她可能花 10 分鐘就能搞定。我不希望每個人遇到同一個問題的時候，都要花半個小時去解決。所以一旦我們用一個斑馬手冊這樣的概念，把所有你的 Claude Code 的記憶串起來以後，她踩過這個坑，下個人再踩的時候，他就只要花 10 分鐘。他不需要去繞那些路，所以我們現在是用這個東西。但他的實現現在還很粗暴，就是真的只有一個 Github 的 Repo，然後每次在用的時候大家先更新一下，然後再慢慢找。這個東西它其實還有很多可以被技術優化的細節在裡面。

對，那這邊其實我剛剛已經稍微提到了嘛，就是說在未來不管你是哪個產業、你是哪一間公司、你是做什麼行業，Agent 是主要的執行者。人的工作其實只是命令 Agent，甚至現在開始有一些產品是讓 Agent 命令 Agent，像剛剛那個豆腐他用龍蝦去命令 Claude Code。對，他現在我在演講的同時，他還在更新這個簡報。對，就像這樣的方法，讓 AI 去調用 AI 已經越來越流行了。所以終端的使用、終端的執行任務的人，已經不再是人類。我不知道大家知不知道 YC？美國有一個加速器叫 Y Combinator。他們以前的 Slogan 叫做 "Make Something People Want"，做一些人們需要的或者是想要的事情。而他們現在的 Slogan 叫做 "Make Something Agents Want"，做一些 Agent 想要的事情、Agent 需要的東西。其實，我們在內部或者是在做產品的時候，也開始往這個方向去想。就是到底這個東西現在是誰在用，未來會是誰在用。如果我們今天做 Knowledge Base 或者做 Wiki，其實以後已經不會有人在看了。其實我們應該可以想一個新的辦法是，乾脆就讓 Agent 看就好了。這個東西人不會去看啊，所以有些東西就不用做得太漂亮。現在在一個內部的概念吧，就是如果以後每個人都透過 Agent 來工作的話，那同步知識應該是讓 Agent 去同步知識，然後 Skills 應該是讓 Agent 之間去同步這些 Skills，對，像這樣子的一個概念。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：斑馬手冊是團隊的「外掛記憶卡」。與其寫給人看的文檔 (People Wiki)，不如寫給 Agent 看的腳本 (Agent Instruction)。
> - **應用**：PM 應負責維護 GitHub 上的「邊界知識」。當 AI 查不到答案時觸發的 Issue，就是產品文檔最優先的補足清單。
> - **連結**：與「檢索增強生成 (RAG)」與「分散式知識管理」關聯。

### 姿勢四：產品結合 Skills 建立迭代閉環

第四個，我覺得是最精彩的地方，也是大家如果你是創業者或者是公司裡面的 PM 或者是創新部門的人，一定要開始嘗試或者是開始了解的東西：是用 Claude Code 的 Skills 去把你的產品跟現在的這些 AI 做一個結合。因為就像剛剛講的，現在你的產品有可能以後是 AI 在用的。所以到底你的產品跟你競品的產品，誰是 Agent 更會用、更好用、更快可以使用到的東西，將會是未來的一個關鍵的差異。

那在這個時候其實， Zeabur 就做了一個什麼事情呢？其實我們現在已經開發了 Zeabur 的這個 Claude Code 的插件。插件裡面有大量關於 Zeabur 的 Skills。這個東西現在還只是一個公開的倉庫，我們還沒有做任何的 PR、任何的宣傳。所以其實只有非常少部分的技術狂熱分子現在已經開始在使用。但是我們會在最近做一些官方的公告，以及一些教學內容，去教大家如何在你的 Claude Code 裡面安裝我們的插件。那一旦你安裝完了以後，其實你會發現 Zeabur 這個東西它跟你以前的想像就完全不一樣了。

怎麼說呢？像我這邊舉個例子。我先跳一下。這邊這個例子呢，是今天我在 Claude Code 開發完了一個東西以後，我甚至連 Zeabur 都不用打開。再講得更極端一點，我連 Zeabur 是什麼都不用知道，我就跟他說幫我部署這個網站。這句話是很多人可能都跟你的 AI 講過的話，有些人可能跟 Google AI Studio 講過這句話。在講這句話的時候，AI 總是在鬼打牆，總是在不知道的地方卡住。以及他在一個能用的技能去幫你完成這個部署的工作。所以我們很多的朋友會聽到他說：「我真的已經用某個 AI 把我的東西給做好了，我也確定本地能跑，但我總是卡在怎麼部署、怎麼上線這件事情。」以前我們 Zeabur 的產品我總是只能貼一個教學影片給他，或者是貼我們的登錄註冊的那個頁面給他，說你去登錄註冊以後這個東西可以幫到你。但那其實不是他要的，因為他整個產品的開發過程就是在 Claude Code 裡面完成的嘛，你還硬要他去學一個新的東西，其實這是很不 AI Native 的事情。所以我們現在就反過來，就讓他最直覺的想法是什麼？問 AI 怎麼部署，那我們就讓 AI 知道怎麼部署就好。

所以當你跟他說「幫我部署這個網站」，對，那他其實就開始幫你研究你現在在 Zeabur 上面是什麼狀態。他發現：「欸，你有很多個狀態，你想要怎麼搞？」然後我就說：「我想要建立一個新專案。」而且這邊有一個很酷的事情是，其實我們是沒有做任何產品化的事情。我們在 Skills 裡面只有說明你作為 AI 怎麼用 Zeabur 的 CLI 而已。可是因為 Claude Code 自己本身是一個已經 Fine-tune 過很多次的產品，所以當他發現你有很多 Zeabur 專案的時候，他會用 Claude Code...你可以用上下鍵去選。

在 Skills 這塊有一個我很想要分享的一個經驗跟技巧，是你要去建立一個「迭代的閉環」。什麼叫做閉環呢？其實講得很簡單，就是說讓 Claude Code 自己去寫自己需要的 Skills。所以比如說我今天想要幫 Zeabur 開發一個部署的 Skill，那我就問 Claude Code 說：「你能不能幫我部署這個東西？」然後他就會開始鬼打牆，因為他不知道怎麼部署，他就哩哩哩哩哩，開始到處亂撞牆。撞到差不多了以後就說：「好，可以了，你為什麼要撞到這麼多人，也不是這樣子就好了嗎，為什麼這麼簡單的事情都不會？」就開始辱罵他。對。那辱罵他以後他就會開始反省，就對，這麼簡單的事情我怎麼不會呢。然後他就會發現說，哦，因為可能某個 Skills 裡面它沒有提到這個東西，所以我不管怎麼樣繞、怎麼樣去查，我就查不到這個東西。所以我就說：「那你是不是可以把這個東西就自己加進去裡面，這樣下一次我問你的時候你就知道要這樣做？」他說：「欸，有道理。」然後就自己加進去。

所以用這樣子的方法，你就會發現它有一個閉環在。就是你只要訂好幾個路徑，以 Zeabur 來舉例，就是部署服務是一條路徑、買機器是一條路徑、綁域名是一條路、然後可能比如說備份資料，這些都是。我用好幾條路徑，我在這個路徑裡面我去執行 Claude Code，試著讓他去完成這個任務。他跑跑跑跑跑，他就會發現卡住，卡住以後他就會發現為什麼我卡住，因為我少了什麼東西所以卡住，他就自己去修那個東西，修完之後他自己就不卡住了。所以我唯一的工作是什麼？定義有哪些任務是你一定不能卡住的。我定義好以後，他就真的不會卡住了。甚至這個定義任務這件事情，我是不是還可以再交給更 High-Level 的人？作為一個懂 Zeabur 的 Claude Code，有哪些任務是你應該要可以很順利的完成？今年列 20 個，這 20 個就進入迭代的閉環，自己去開始安裝、自己去開始測試這些 Skills。

剛剛提的那個點就是「引導反省」這個事情。它的 Prompt 有一個我實驗下來我覺得最好用的 Prompt 是這個。在研究讓你的 Claude Code 去幫你修你的 Skills 的時候，有一個關鍵的 Prompt 是：**作為一個開發 Skills 的開發者，你為什麼會遇到這個問題？** 你一定要加這句話喔，否則你在罵他的時候他只會很難過而已，很難過。就是我會說：「為什麼這麼簡單的事情不是這樣就好了嗎？你怎麼都不會？」他就說：「抱歉你說的對，讓我用新的方法重新再試一次。」我說：「不，我不是要你真的用新的方法試一次，我是要你檢討為什麼你這一次沒有成功。」對，所以這邊有一個案例。

這個是熱騰騰的案例，是我今天在讓我的 Claude Code 幫我去部署一個 MongoDB。他就直接報錯兩次。報錯兩次之後他開始嘗試新的方法，然後就用一個我根本就不知道那是什麼的一個指令開始在亂拔亂拔亂拔，拔到這邊我就看不下去了，我就直接暫停。我說：「你為什麼會用這種已經被淘汰的方法來部署呢？是哪個 Skills 或哪個 CLI 沒有寫清楚嗎？請你用 Skills 的開發者的角度分析你前面這邊到底在幹嘛？」然後他就開始說：「好問題，讓我來研究一下。」然後他其實真的很快，大概不到 30 秒的時間，他就直接出來了。說研究完之後發現，沒有任何一個 Skill 涵蓋了部署 Marketplace 服務的這個場景。對，那後來我就...你看我幫忙補一下 Background。是在 Zeabur 裡面，我希望他做的是去部署模板，這個 Skill 裡面幫我部署一個 MongoDB 的模板。可是因為我剛剛的 Prompt 是「幫我部署 MongoDB」，有提到「模板」這個詞，所以他就沒有去 Load 那個部署模板的 Skills 進來。所以他需要用到的所有知識他都沒有用到。

那我讓他檢討過了以後，他馬上就發現說，因為沒有任何 Skills 被你命中，所以我就沒有去看到那些知識。沒有看到那些知識怎麼辦呢？我就會模型回退到我原本就有的那些以前的訓練知識。我那個訓練知識都是去年或者是前年的東西，所以他最後就用了一個很古老、已經被淘汰的方法來部署。所以最後他就發現說，建議你作為開發者可以新增一個這個東西，核心問題是什麼。那下一句你的 Prompt 是什麼？你的 Prompt 就是：「那你自己去修就好了，不要來建議我了。你自己去那邊把這個東西修好就好。」這樣子他就可以自己把自己的問題給解決掉。

假設有人說：「有沒有可能我可以讓這個 Claude Code 可以看 Dcard？」他自己上去 Dcard 看一下最近的新聞是什麼，然後自己把它總結給使用者。但是像這樣子的一個 Idea 的話，他其實也可以用相同的方法，自己去做這個迭代的閉環。他的唯一的 Prompt 就是：「你去試試看能不能從 Dcard 那邊幫我找到新聞。」這個 Prompt 你下去以後，他自己跑完這個迭代的閉環，下一次你問順便就能完成。

如果你是在做軟體開發的話，你可以用這個。如果我不是軟體開發呢？其實是一樣的例子。我舉個非常極端的例子。比如說，大部分工程師如果想創業的話，一般來說最後想創業的 Idea 是開一間咖啡廳。因為工程師喜歡在咖啡廳工作，所以有一間咖啡廳的話是人生的終極目標。這個時候如果我開了一間咖啡廳，我就可以跟 Claude Code 說：「幫我買一杯咖啡。」你會發現他會開始到處鬼打牆。作為網絡上的 AI，它如何都能幫我買咖啡？如果能建立自我迭代的閉環，它就會發現，如果一家咖啡廳的公司或是平台，把它的服務先寫成 API，包裝成 CLI，我的 Skills 下如何的引導，我好像就可以有辦法幫我的主人、幫我的使用者去買咖啡。說不定它就真的能夠在...而且我覺得大概半天以內，它就真的能幫咖啡廳建立好這個連結。我覺得它都是未來有辦法去接的東西。當然不只是接 Claude Code，你也可以接各種其他的東西。

其實我剛在輸出一大堆的時候，我講什麼 API、講什麼 CLI、MCP 什麼的東西的話，其實我之前發了一篇 Threads，是一個超級長文。組合而成的一個 Threads，然後它就是在講這個東西。我會希望今天我在講這個東西的受眾不是工程師嘛。最後就是用一頁快速的總結一下。我覺得在現在這個時代，Claude Code 絕對不會只是工程師的工具。我覺得是任何人，哪怕你是非技術的人員，你只要能夠掌握，而且你一定能夠掌握，你可以在你的工作跟你的公司、你的團隊裡面發揮非常非常多的價值。開發工具成為現在最主流、直接統治整個開發市場的原因，因為這個生態真的是改變了讓你的 AI 學會一件事情的一個關鍵的生態差別。對。而且更重要的是，如果你能夠共享你的知識庫的話，其實你能夠把一個人的成果共享給你整個團隊。你教會你的 AI 做了一件事情，這叫一個成果，把這個成果共享給你整個團隊，其他人就不用再花時間去教那個 AI 了。對。那這是很重要的事情。最後就是你試著把你的產品做成一個 Skill，讓你的產品不只是做給人用的，而且是做給 AI 用的。因為這是未來的趨勢。所有的服務、所有的產品，都是被 AI 使用，而不是被人使用。對。這就是最後面，謝謝大家。謝謝大家。

# 第二部分：QA 實錄 (臺北大巨蛋.srt)

> **註**：本部分為演講後的現場問答。

## Q1：關於 Local First 與知識庫同步的機制

我們會期待這個知識庫變成所有人的 Claude Code 最高頻會使用的一種 Skill，這是肯定的。我們甚至都希望說，不管我問了什麼，只要他一提到 Zeabur 這個關鍵字，他就一定要去翻一下，把他拿走。在這個情況下，他一定必須要是 Local First。也就是說，哪怕我在一個網路很慢的環境，他也可以不要去線上刷 API 的東西，直接用現有的知識去回答你。也就是說我們希望這個知識庫它是可以 Local First，但有一個線上的東西作為一個 Truth 去同步他的事情。那如果我們想要去做這個同步的事情，其實 Git 是最簡單的同步的事情。所以我們為什麼會把它放在 GitHub？是因為你放在 GitHub 以後，其實它會去 Pull 東西下來。Pull 東西下來以後，我每次在問它問題的時候，它是從我的本地去找東西。找完了以後，它下一次才會偷偷在你沒有用的時候去更新一下，更新回來。給你反應說我下一次再問，它就會馬上是最新的。對，這是第一個問題。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：這就像是從「每次都查線上字典」進化到「手機離線字典，但有網路時會自動下載新字」。
> - **應用**：PM 應評估團隊工具在弱網環境下的表現。將知識庫 Local First 化，能極大提升 AI 的反應速度與穩定性。
> - **連結**：與「邊緣推理 (Edge Inference)」與「離線優先架構」關聯。

## Q2：自我迭代閉環的模型選擇與盲點

然後第二個問題是你說，迭代，就是自我迭代的那個閉環，有沒有可能用其他的 AI 來去用，會不會效果更好？其實這肯定效果會...不一定說更好。因為我們在 Claude Code 裡面用的是 Claude 3.5 Sonnet，是現在在 Coding 裡面效果、就是 Thinking 的效果最好的模型。但是你雖然用的是最好的模型，用自己檢查自己會有一個可能的問題是盲點的問題。因為同個模型它的訓練數據是一樣的，它如果訓練數據少了一部分的話，它永遠不會繞進那一塊去做思考。所以你想說我換一個模型來去檢查你自己，是可以避免到這個盲點的問題。但同時要考慮的是，我換的那個模型會不會其實比 Opus 弱很多之類的。這是第一個問題。第二個問題是我在 Claude Code 裡面，它不需要任何額外的設定，它就可以知道自己剛剛犯了什麼錯。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：就像是「自己校對自己的文章」vs「請編輯校對」。雖然換個腦袋好，但如果編輯程度太差反而會改錯。
> - **應用**：PM 在規劃 AI 監控流程時，應考慮「模型交叉驗證」的成本與收益比。
> - **連結**：與「模型對抗」或「LLM-as-a-Judge」概念關聯。

## Q3：資安疑慮：公司知識庫的洩漏風險

把你好全部 Zeabur 的知識都弄上去，那你會不會有在意，就是萬一你的公司的商業秘密或段落佈局被外面知道？就是我只要講一下，你有做任何的考量或什麼的處理？

OK，那這個東西它其實本質上回歸到一個原始的問題，是我們的這些 Knowledge 是怎麼儲存的。那就跟剛剛這位先生有問到的一樣的問題。因為如果我們的斑馬手冊會洩漏出去，代表我們的要嘛是 GitHub 上面的權限環境有問題，要嘛是你本地的電腦有問題。這兩個地方出了問題導致它洩漏出去。那如果真的本地出了問題的話，駭客要偷的絕對不是斑馬手冊，要偷的絕對是我們的程式碼、原始碼本身。所以我們相當於是用以前在考慮這個問題的資安方法，去套在我們現在的一個新的應用。所以這邊也剛好回復剛剛那個問題，為什麼要選一套，其實也是因為它的生態已經很完整。我們不用額外去思考說，我如果用 Notion 的話，會有什麼設定沒有設定好，然後導致這個東西會洩漏出去等等的。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：與其打造一個新的「保險箱」，不如把財寶鎖在已經驗證過數十年的「銀行金庫」（GitHub Private Repo）裡。
> - **應用**：PM 在推動 AI 工具落地時，應優先複用現有的資安框架 (Security Framework) 以降低合規阻力。
> - **連結**：與「成熟度模型」與「最小特權原則」關聯。

## Q4：如何驗證 Skill 的價值與 Discovery 機制

第一個問題是，如果我今天想要把我的產品把它 Skill 化，讓用戶可以安裝取用的話，我有什麼方法可以驗證我的 Skill 夠好用，確保用戶在裝的時候真的有幫助？第二個問題是，用戶要怎麼樣知道說，你的這個產品有 Skill 可以安裝？

OK，好。這兩個問題問得非常好，非常精準。我們的插件已經上線了，可是我不去講，就是沒有人知道。Claude Code 目前也不會自動推薦。現在還是依賴兩大事情。第一個是，你要讓使用者自己有一個跨過這個啟動的檻。就是他要先知道「我要部署」，然後一搜尋 "Zeabur Skills" 就會看到如何安裝。第二個事情是，如果說 Zeabur 這個東西，它已經能夠被訓練到它的 Base Model 裡面。最近有人跟我分享，他在沒聯網的情況下問 AI 如何部署，AI 突然出現了 Zeabur 的名字。這代表模型已經收集到 Zeabur 並將其訓練進去了。

關於如何確保 Skill 夠好：這也是我很苦惱的一個。關鍵在於：1. 它有沒有辦法命中你想要他命中的那個 Skill。2. 處理 Skill 的相依性。例如「部署」Skill 裡面如果沒有關於「建立專案」的知識，他就永遠建立不了專案。所以這也是 Skill 產品化之後，我們要去解決的邏輯鏈條問題。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：AI 時代的 SEO 變成了「Base Model 訓練權重」與「觸發詞命中率」。Skill 的相依性就像是「產品導覽 (Onboarding)」的流程設計。
> - **應用**：PM 在定義 Skill 時，應專注於「原子功能」的組合邏輯，確保每個 Skill 都能引導 AI 進入下一個動作。
> - **連結**：與「搜尋引擎優化 (SEO)」與「AI 代理介面 (Agentic UI)」關聯。

## Q5：知識庫的維護與 Review 機制

提問：關於知識庫補完的過程，是請 AI 補還是人補？補完後是由 AI 去 Review 還是人類 Review？

這又繞回到第一個問的問題了。知識庫裡的每條知識，在 Zeabur 的系統裡面就像是程式碼一樣。所以，那是誰在 Review 一個程式碼？Zeabur 現在在用的是一個叫 CodeRabbit 的工具。我們不管是前端、後端，還是 Skills，還是我們斑馬手冊的知識庫，每個 PR 都是經過 CodeRabbit 的審核。我很期待 Claude Code 的原生 Review 功能有沒有辦法做得更好。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：從「人工校閱」進階到「自動化合規檢查」。這確保了知識庫的品質不會隨著參與人數變多而下降。
> - **應用**：PM 可以設定知識庫更新的「AI 閘門」，確保所有新增知識都符合產品邏輯。
> - **連結**：與「持續整合 (CI)」與「代碼質量管理」關聯。

## Q6：斑馬手冊的行動端維護

提問：有辦法透過手機去進行斑馬手冊的維護嗎？

因為我們選擇用 GitHub 作為載體。所以任何一個能夠碰到 GitHub 的 AI Agent，都能幫我更新斑馬手冊。所以哪怕你是用手機上的 Claude App、Gemini，只要你能訪問 GitHub，就能幫我們去做這個斑馬手冊的迭代跟補充。

> 🎯 **產品經理 (PM) 視告**
>
> - **類比**：只要有「鑰匙」(Git Access)，不論在哪個「門」(Device) 都能進入。
> - **應用**：PM 在碎片化時間產生的靈感，可以透過語音轉文字餵給行動端 Agent 提交 PR。
> - **連結**：與「跨平台工作流」關聯。
